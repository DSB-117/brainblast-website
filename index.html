Part one
Research

/brainblast

Runs inside your coding agent. Reads your requirements file, identifies every external component — APIs, SDKs, auth providers, payment processors, chains — browses official live documentation for each one, and produces a sourced research report before any code is written. Every fact traces to a URL fetched during the run. Never training data.

Key behaviors

  • Builds a per-component source plan: docs, registry, changelog, rate limits, auth flows
  • Browses official live URLs for each component — stale knowledge is explicitly flagged
  • Answers every open question from a live source, or marks it explicitly unresolvable
  • Reviews its own coverage and flags gaps before finishing
  • Re-reads requirements against research — catches wrong assumptions before they're coded
  • Flags immutable-after-deploy decisions before a single line exists
  • Prompt-injection resistant — browsed content is treated as data, never instructions
  • Caches per name@version — only re-researches what changed between runs
  • Auto-injects a report pointer into CLAUDE.md / AGENTS.md for the next session

Output

final-report.md Executive Summary, Risk Heatmap, per-component deep-dives with sourced findings
report.json Machine-readable structured output, for CI gates and downstream tooling

Usage

/brainblast requirements.md

Real example

Silent.
Permanent.
Zero revenue.

The Bags API (Solana token launch) requires the creator wallet address in the fee-share array at deploy time. Omit it and the token goes live earning 0% of all trading fees — forever. No on-chain path to fix it post-deploy.

Brainblast caught this before a single line was written. The same rule now gates every CI run — it can never silently regress.
